1. High-Level 6-Stage Pipeline End-to-end data flow

Market data enters from the left (Stage 1), flows through feature computation, alpha generation, risk validation, and execution, then exits as exchange orders on the right. The return path (fills, balance updates) feeds back through the EventBus. The RebalanceFSM autonomously drives Stages 3-5 on a timer or signal basis.

2. EventBus Architecture Pub/sub wiring

All market data events flow through a single in-process EventBus. Publishers (client manager workers, user data stream, RebalanceFSM) emit typed frozen events. Subscribers (strategy hooks, portfolio) receive them sequentially with natural backpressure. Error isolation via try/except at each dispatch wrapper ensures one failed subscriber cannot block others.

3. RebalanceFSM-Driven Pipeline Stages 3-5 via compute -> validate -> execute -> cooldown

The RebalanceFSM replaces direct submit_weights() calls with an explicit state machine. On each timer tick or request_rebalance() signal, it drives the full cycle: compute_weights() → optimizer.validate() → execution_engine.execute() → cooldown. Each stage has a dedicated callback; failures and empty signals route to appropriate recovery states.

4. Risk Constraint Pipeline PortfolioOptimizer.validate() flow

Each constraint step projects the weight vector onto a feasible set. Steps execute sequentially — the output of one feeds into the next. The optimizer is a stateless constraint projector, not a Markowitz optimizer. RiskConfig is intentionally mutable so limits can be tightened at runtime (e.g. during volatility spikes).

5. Execution Engine Detail Weights to exchange orders

The ExecutionEngine converts validated weights into concrete order intents, then submits them to exchange connectors. Sells execute before buys to free capital. Each order is submitted independently with error isolation — one failure does not block others.

7. Multi-Venue Multiplex Socket Architecture Market data ingestion

Each venue uses a single WebSocket per data type (kline/OB), carrying all symbols on one TCP connection. Messages are queued and processed by async worker pools. The system supports 4 venue x data-type combinations: Binance Spot, Binance Futures, Aster Spot, and Aster Futures — each with a kline and OB client manager (8 total), plus dedicated user-data managers for Binance Spot and Futures (10 managers total). Binance Spot user data uses the new WebSocket API (wss://ws-api.binance.com:443/ws-api/v3) with HMAC-SHA256 signing via userDataStream.subscribe.signature (the deprecated listenKey endpoint /api/v3/userDataStream was retired on 2026-02-20). Binance Futures user data still uses the listenKey approach (POST /fapi/v1/listenKey + keepalive every 30 minutes), which remains current for USDS-M Futures.

8. Multi-Strategy: WeightAggregator + ShadowBook Capital allocation and per-strategy attribution

When multiple strategies share one portfolio, a WeightAggregator merges their weight vectors — each scaled by its capital allocation fraction — into a single combined vector for the optimizer. Each strategy owns a ShadowBook (virtual position ledger) so strategies can read their own isolated positions, cash, weights, and PnL without touching the aggregate Portfolio. ShadowBooks are updated in real time via event-driven fill attribution: each ORDER_UPDATE event is routed to the correct ShadowBook using ExecutionEngine._order_strategy_map. After order submission, lock_for_order(order_id, intent) is called on the ShadowBook to reserve locked cash/qty for pending LIMIT orders. Locks are released incrementally on each partial fill and fully on terminal status via _partial_release_lock() and release_order_lock(). Note: WeightAggregator.attribute_fills() is a documented no-op. It previously called reconcile_shadow_books() which overwrote event-driven fills with mark-price approximations. Fill attribution is now handled entirely in real time via ORDER_UPDATE events routed through _order_strategy_map.

8b. Sub-Account Mode vs Shared-Account Mode — Full Wiring Topology register_strategy() two-path fork

StrategyRunner.setup_strategy() forks at _has_sub_account(strat_cfg) (returns True when strategy_config.sub_account_api_key is non-empty). The two paths differ in exchange ownership, EventBus wiring, ShadowBook role, and aggregator participation. Path A (sub-account) gives a strategy full autonomy over its own account; Path B (shared-account) lets multiple strategies co-operate over a single shared exchange connection via the WeightAggregator.

EventBus Split in Sub-Account Mode

The key architectural insight of sub-account mode is the split EventBus subscription: market data (KLINE, orderbook) flows through the shared unauthenticated bus (shared across all strategies and therefore cheaper — one WebSocket connection per symbol), while user data (fills, balance changes) flows through a private isolated bus backed by the strategy's own user data stream. This guarantees that a sub-account strategy never sees fills or balance events from other strategies.

9. StrategyRunner Startup Sequence Initialization order

The StrategyRunner orchestrates all component initialization in a strict dependency order. It progresses through RunnerState: CREATED → CONFIGURING → CONNECTING → SYNCING → RUNNING → STOPPING → STOPPED. The pipeline (Portfolio → Optimizer → ExecutionEngine) is set up in the SYNCING phase, after the exchange is authenticated and balances are fetched.

12. Concurrency Model Single event loop, everything async

All I/O, event processing, and pipeline execution share a single asyncio event loop. Natural backpressure comes from EventBus.publish() awaiting callbacks sequentially. CPU-heavy work is offloaded to a ProcessPoolExecutor via strategy.run_heavy(fn). PeriodicTask replaces bare while True + sleep loops for recurring operations.

14. Recent Changes & Known Limitations As of 2026-03-30

Summary of material implementation changes since the previous diagram revision, plus known architectural issues that have been identified but not yet resolved.

Recent Implementation Changes

ComponentChangeDetail
BinanceUserDataClientManager Full replacement — new Binance WebSocket API Now connects to wss://ws-api.binance.com:443/ws-api/v3 and authenticates via userDataStream.subscribe.signature (HMAC-SHA256). The old listenKey endpoint /api/v3/userDataStream was retired on 2026-02-20. Message envelope changed: events arrive as {"subscriptionId": N, "event": {...}}.
BinanceFuturesUserDataClientManager Newly added class Manages USDS-M Futures user data stream. Uses POST /fapi/v1/listenKey to obtain a key, connects to wss://fstream.binance.com/ws/<listenKey>, and sends PUT /fapi/v1/listenKey keepalive every 30 minutes. Dispatches ACCOUNT_UPDATE and ORDER_TRADE_UPDATE events.
WeightAggregator.attribute_fills() Now a documented no-op Previously called reconcile_shadow_books() which computed fill attribution from mark prices, overwriting the more accurate event-driven fills. Fill attribution is now entirely event-driven via ORDER_UPDATE events routed through ExecutionEngine._order_strategy_map.
TargetWeights.strategy_id Type changed str → int (default 0) Aligns with OrderIntent.strategy_id: int and ShadowBook routing which always uses integer keys in _order_strategy_map.
Feed __call__ methods Now no-op stubs Feed lifecycle (start, stop, reconnect) is owned entirely by run_strategy.py via the client managers. Feeds no longer self-start when called. Any diagram section that showed feeds opening their own WebSocket connections is now incorrect.
Portfolio._on_balance Now handles non-stablecoin assets When a BALANCE_UPDATE event carries a raw base asset (e.g. "ETH") that is not a stablecoin, the handler now maps it to the trading pair (e.g. "ETHUSDT") using exchange._token_infos before creating or updating the position entry.
_create_sub_account_exchange() Now passes args=self.cfg The BinanceSpotExchange constructor called for sub-account strategies now receives the full AppConfig via args=self.cfg, enabling it to read risk/portfolio config the same way as the primary exchange.
cancel_all_orders Bug fix: uses order.id instead of order['order_id'] Previously accessed the order object as a dict; now correctly accesses the Order dataclass attribute .id.

Known Limitations (Identified, Not Yet Fixed)

IssueLocationImpact
_completed_order_ids is an unbounded set() Portfolio._completed_order_ids, ShadowBook._completed_order_ids Grows indefinitely over the lifetime of the process. In a high-frequency or long-running deployment, this will cause steady memory growth. A bounded LRU set or TTL-based eviction should be applied.
