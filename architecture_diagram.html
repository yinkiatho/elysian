1. System Overview UPDATED Top-level component topology

Elysian now runs as a multi-container Docker system. A dedicated MarketDataService container owns all WebSocket feed managers and publishes market data events to a Redis pub/sub broker. Each strategy runs in its own isolated container, subscribing to the Redis channels relevant to its symbols. This replaces the former single-process model where one StrategyRunner managed both feeds and strategies in the same asyncio event loop.

Key Changes from Previous Architecture

  • MarketDataService — new dedicated container that owns all WebSocket client managers (BinanceKlineClientManager, BinanceOrderBookClientManager, etc.)
  • Redis pub/sub — new network boundary between market data and strategy layers; replaces in-process EventBus for market data distribution
  • One container per strategy — elysian_strategy_N containers each run one strategy, subscribing to Redis for their symbols
  • Private EventBus still in-process — account events (ORDER_UPDATE, BALANCE_UPDATE, REBALANCE_COMPLETE) remain on the in-process private EventBus inside each strategy container
  • Two-tier YAML config — trading_config.yaml (system) + per-strategy YAML files (strategy_NNN_*.yaml)

3. MarketDataService — Feed Layer NEW

MarketDataService (elysian_core/services/market_data_service.py) is the dedicated feed container. It instantiates all WebSocket client managers, registers feeds for every configured symbol, and injects a RedisEventBusPublisher as the event bus for each manager so that market data events are serialised and published to Redis. It does not run any strategies, sub-account exchanges, or risk/execution components.

EventBus interface compatibility RedisEventBusSubscriber implements the same subscribe() / unsubscribe() / publish() interface as the in-process EventBus. Strategies call shared_event_bus.subscribe(EventType.KLINE, cb) identically regardless of whether the bus is backed by Redis or an in-process deque. publish() on the subscriber side is a no-op — strategies never publish to the shared bus.
Serialisation Events are serialised as JSON via _EventEncoder (handles datetime, enums, and SortedDict). OrderBook bid/ask levels are serialised as lists of [price, qty] pairs and reconstructed into a plain OrderBook (not a BinanceOrderBook or AsterOrderBook) on the subscriber side. This is intentional — only the level data matters for mark-price and signal computation.

5. 5-Stage Rebalance Pipeline Data flow through a single rebalance cycle

The six-stage pipeline is unchanged in structure. The only architectural difference is that Stage 1 (market data) now runs in a separate container and delivers events to Stage 2 (strategy dispatch) via Redis rather than in-process. Stages 3–5 remain sequential direct method calls within the strategy container.

It reads STRATEGY_CONFIG_YAML from the environment, constructs a RedisEventBusSubscriber as the shared event bus, and no longer creates any WebSocket client managers (those live in the MarketDataService container).

7. Strategy Event Bus Wiring UPDATED Shared vs private buses across two containers

The two-bus architecture is preserved but now the shared bus is backed by Redis while the private bus remains in-process. The strategy subscribes to 5 event types across the two buses. The ShadowBook also subscribes to KLINE on the shared bus for mark price updates. ORDER_UPDATE and BALANCE_UPDATE are dispatched by the strategy to the ShadowBook via asyncio.gather() rather than ShadowBook subscribing directly.

_asset_symbol_map and _position_deducted_comm (new fields) _asset_symbol_map maps raw base asset names (e.g. "BNB") to trading symbols (e.g. "BNBUSDT"), populated whenever apply_fill() is called with a base_asset. This allows non-stablecoin commission assets (e.g. BNB fee on an ETHUSDT trade) to be resolved to their position and have their qty deducted, mirroring the real exchange balance. _position_deducted_comm tracks these position-side deductions so they are excluded from total_commission() in USDT to avoid double-counting in net_pnl().

12. RiskConfig Constraints PortfolioOptimizer validation pipeline

The RiskConfig drives Stage 4 of the rebalance pipeline. The optimizer now runs a 7-step ordered constraint pipeline (previously 6 steps), adding an input validation step at the front to drop NaN / Inf / non-numeric weights before any constraint is applied.
