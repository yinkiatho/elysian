1. System Overview Top-level component topology

StrategyRunner is the root orchestrator. For each configured strategy it wires together an exchange connector, a set of data feeds, two event buses (shared market data + private user data), a ShadowBook ledger, a PortfolioOptimizer, and an ExecutionEngine. The thin Portfolio object aggregates NAV across all shadow books without owning any raw exchange state.

2. 5-Stage Rebalance Pipeline Data flow through a single rebalance cycle

A single rebalance cycle touches five distinct stages. Each stage has a well-defined input and output type, making the pipeline easy to test in isolation. Stages 1–2 are continuous (streaming); Stages 3–5 are triggered by the RebalanceFSM on a timer or a manual request_rebalance() call.

Design decision: sells before buys Stage 5 always submits SELL orders ahead of BUY orders to free up quote balance before attempting purchases. This prevents unnecessary order rejections due to insufficient free cash in accounts that hold no separate credit facility. The 0.001 weight-delta threshold eliminates micro-trades whose transaction costs would exceed the rebalancing benefit.

3. Order Data Model Hierarchy src/models/order.py

Orders follow a strict inheritance chain. The base Order holds the canonical exchange fields. ActiveOrder adds fill-delta bookkeeping for streaming partial fills. ActiveLimitOrder further adds reservation accounting so the ShadowBook can lock cash or base-asset qty the moment a limit order is submitted — before any exchange acknowledgment arrives.

last_fill_price captures the Binance field "L" — the price of the most recent incremental fill, distinct from avg_fill_price. sync_from_event in ActiveOrder returns the quantity delta (new fill minus previous fill) so callers can apply exactly one incremental accounting update per WebSocket message without re-processing prior fills.

4. Position & ShadowBook Per-strategy virtual ledger

The ShadowBook is the single source of truth for a strategy's financial state. It mirrors the real exchange state through event-driven updates rather than polling, and exposes a rich read API (NAV, weights, drawdown, exposure) that the strategy and optimizer consume. Portfolio is a thin read-only aggregator — it holds references to shadow books but never mutates them.

Stablecoin seeding On initialisation, _mark_prices seeds all detected stablecoins (USDT, USDC, BUSD) at 1.0. This ensures NAV calculations remain correct when the strategy holds a mix of stable and volatile assets before the first kline event arrives.
Locked cash vs free cash _locked_cash is the running sum of all ActiveLimitOrder.reserved_cash for pending BUY limit orders. free_cash = _cash - _locked_cash. The optimizer uses free_cash (not gross cash) when sizing new orders, preventing double-allocation.

5. Signal Contract Flow Type-safe handoffs between pipeline stages

Each stage boundary is a typed dataclass, making the full pipeline statically verifiable. The sequence below shows one complete rebalance cycle from the strategy emitting weights to the completion event being delivered back.

sequenceDiagram participant STRAT as SpotStrategy participant FSM as RebalanceFSM participant POpt as PortfolioOptimizer participant EE as ExecutionEngine participant CONN as SpotExchangeConnector participant SB as ShadowBook participant PBUS as "Private EventBus" STRAT->>FSM: request_rebalance(**ctx) OR timer_tick fires FSM->>STRAT: compute_weights(**ctx) STRAT-->>FSM: TargetWeights{weights, timestamp, strategy_id, venue, metadata, price_overrides} FSM->>POpt: validate(TargetWeights) POpt-->>FSM: ValidatedWeights{original, weights, clipped, rejected, rejection_reason} FSM->>EE: execute(ValidatedWeights) EE->>EE: snapshot mark prices from feeds EE->>SB: total_value() SB-->>EE: nav: float EE->>EE: compute delta_qty per symbol EE->>EE: round to step_size / filter threshold loop For each SELL intent EE->>CONN: place_limit_order(symbol, SELL, price, qty, strategy_id) CONN-->>EE: order_id: str EE->>SB: lock_for_order(order_id, intent) SB-->>EE: ActiveLimitOrder created (reserved_qty locked) end loop For each BUY intent EE->>CONN: place_limit_order(symbol, BUY, price, qty, strategy_id) CONN-->>EE: order_id: str EE->>SB: lock_for_order(order_id, intent) SB-->>EE: ActiveLimitOrder created (reserved_cash locked) end EE-->>FSM: RebalanceResult{intents, submitted, failed, timestamp, errors, submitted_orders} FSM->>PBUS: publish RebalanceCompleteEvent{result, validated_weights, timestamp, strategy_id} PBUS->>STRAT: on_rebalance_complete(event)

Note that ShadowBook.lock_for_order() is called synchronously within the execution loop, immediately after the exchange acknowledges the order ID. This ensures the locked cash / qty is accounted for before the next rebalance cycle can begin, even if the WebSocket fill event arrives late or out of order.

6. Event System Shared vs private buses, producers and consumers

The system uses two logically separate event buses. The shared bus carries market data visible to all strategies simultaneously. The private bus carries user-data events (fills, balance changes) scoped to a single strategy's sub-account connection. Separating the buses prevents one strategy's order activity from leaking into another strategy's event handlers.

7. Connector & Feed Layer Exchange connectivity and WebSocket client managers

All exchange interactions go through a two-tier abstraction: the SpotExchangeConnector base defines the order-management contract, while AbstractDataFeed defines the market-data contract. Concrete implementations (Binance Spot, Binance Futures, Aster) override only the venue-specific details. WebSocket lifecycle is managed by dedicated client-manager classes, keeping reconnection logic out of the feed business logic.

9. RiskConfig Constraints PortfolioOptimizer validation pipeline

The RiskConfig drives Stage 4 of the rebalance pipeline. All six constraints are applied in a fixed order; earlier steps may eliminate symbols before later steps run. The final ValidatedWeights object records which symbols were clipped or rejected and why.

RiskConfig Field Reference

Field Type Example Description
max_weight_per_asset float 0.50 Maximum fraction of portfolio NAV in any single symbol. Applied per-asset before exposure scaling.
max_gross_exposure float 1.00 Maximum sum of absolute weights. Values below 1.0 enforce a cash buffer at the portfolio level.
min_cash_weight float 0.00 Minimum fraction kept as stablecoin cash. Prevents the portfolio from being fully invested.
max_turnover float 0.20 Maximum portfolio turnover per cycle (sum of absolute weight changes). Limits transaction costs.
min_rebalance_interval_s float 60.0 Rate limiter — minimum seconds between consecutive rebalance cycles for this strategy.
whitelist Set[str] {"BTCUSDT", "ETHUSDT"} If non-empty, only symbols in this set may receive non-zero weight. Empty set = all allowed.
blacklist Set[str] {"LUNAUSTC"} Symbols explicitly forbidden from receiving any allocation, regardless of strategy output.

10. Database & Persistence Peewee ORM snapshots

Portfolio state is persisted asynchronously to a SQLite (or Postgres) database via Peewee ORM. Both per-strategy ShadowBook snapshots and aggregate Portfolio snapshots are written. The PeriodicTask ensures at most 60 seconds of state can be lost on crash; manual calls to save_snapshot() are also made at shutdown.
