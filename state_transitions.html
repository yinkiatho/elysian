1. RebalanceFSM core/rebalance_fsm.py · core/fsm.py

The RebalanceFSM is the central state machine that drives the full rebalance pipeline. It extends BaseFSM and is purely state-driven — the strategy triggers it by calling request(). Re-entrant transitions are queued and drained after the current callback completes. Context flows as **ctx through every stage, accumulating keys: target_weights, validated_weights, rebalance_result.

BaseFSM re-entrancy: If a callback fires another trigger() while a transition is in progress, it is queued in _pending and drained sequentially after the current callback returns. This prevents deadlocks from callbacks that publish events.

2. Order Lifecycle FSM core/portfolio.py · core/shadow_book.py · core/order.py

Each submitted order moves through a lifecycle tracked by both Portfolio and ShadowBook via _update_order(). The FSM uses Order.transition_to() and Order.update_fill() for fills. Partial fills accumulate via _fill_tracker (order_id → last known cumulative filled_qty) so each ORDER_UPDATE event applies only the incremental delta.

3. BinanceUserDataClientManager — Connection States connectors/BinanceDataConnectors.py

The user-data WebSocket connection manager implements an implicit connection FSM inside _reader_loop(). It uses the new Binance WebSocket API (wss://ws-api.binance.com:443/ws-api/v3) with HMAC-SHA256 signing via userDataStream.subscribe.signature, replacing the deprecated listenKey approach retired 2026-02-20. On any connection error it reconnects with exponential backoff capped at 60s.

Source: elysian_core/connectors/BinanceDataConnectors.py — class BinanceUserDataClientManager
