OrderFSM — Order Lifecycle

elysian_core/core/order_fsm.py  |  elysian_core/core/order.py

Overview

The OrderFSM manages the complete lifecycle of a single exchange order from submission through to a terminal state. It operates in warn-mode: invalid transitions are logged but still applied, because the exchange is the authoritative source of truth for order state. The OrderFSM class wraps a _VALID_TRANSITIONS dict and is instantiated per-order. Module-level functions validate_order_transition() and is_terminal() are provided for backward compatibility.

Warn-mode: If an unexpected state transition arrives from the exchange feed (e.g. FILLED arriving on a CANCELLED order), the FSM logs a warning and still applies the transition rather than raising. The exchange is always authoritative.
Active states
PENDING, OPEN, PARTIALLY_FILLED, TRIGGERED
Terminal states
FILLED, CANCELLED, REJECTED, EXPIRED
Key properties
is_active, can_fill, is_conditional, is_terminal
Fill tracking
sync_from_event() via _prev_filled delta

RebalanceFSM — Rebalance Cycle

elysian_core/core/rebalance_fsm.py

Overview

The RebalanceFSM drives the full signal-to-execution pipeline. Starting from IDLE, it orchestrates weight computation, validation, execution, and cooldown in a strict ordered cycle. A wildcard "suspend" trigger from any state enters SUSPENDED, implementing a drawdown breaker or maintenance pause.

Re-entrancy guard: BaseFSM queues transitions that arrive while one is already in-progress using a deque. This prevents lost events when callbacks trigger further transitions synchronously.
Pipeline stages
IDLE → COMPUTING → VALIDATING → EXECUTING → COOLDOWN
Error recovery
ERROR + "retry" → IDLE (auto-retry after cooldown_s)
Suspend trigger
Wildcard from any state → SUSPENDED
Cooldown
asyncio.call_later(cooldown_s); immediate if 0

StrategyState — Strategy Lifecycle UPDATED

elysian_core/core/enums.py  |  elysian_core/strategy/base_strategy.py

Overview

StrategyState tracks the lifecycle of a running trading strategy from creation through to graceful shutdown. The architecture is unchanged in state names, but STARTING now subscribes across both the shared Redis-backed bus and the private in-process bus. Critically, ORDER_UPDATE and BALANCE_UPDATE dispatched to the ShadowBook happen via asyncio.gather() in the strategy's dispatch methods rather than via a ShadowBook subscription — giving the strategy user hook and the ShadowBook handler concurrent execution.

Changes from Previous Architecture

  • Shared bus is now Redis-backed — _shared_event_bus is a RedisEventBusSubscriber; same interface, different transport
  • 5 subscriptions unchanged — KLINE + OB on shared bus; ORDER_UPDATE + BALANCE_UPDATE + REBALANCE_COMPLETE on private in-process bus
  • _dispatch_order uses asyncio.gather() — on_order_update() and shadow_book._on_order_update() run concurrently per fill event
  • _dispatch_balance uses asyncio.gather() — same concurrent pattern for balance updates
  • _dispatch_kline filters by strategy symbols — checks event.symbol in strategy_config.symbols before dispatching
Shared bus type
RedisEventBusSubscriber (market data from Redis)
Private bus type
EventBus (in-process, account events only)
Fill dispatch
asyncio.gather(on_order_update, shadow_book._on_order_update)
RUNNING trigger
_mark_running() on first dispatched kline event

RunnerState — StrategyRunner Lifecycle UPDATED

elysian_core/core/enums.py  |  elysian_core/run_strategy.py

Overview

RunnerState represents the top-level lifecycle of a single-strategy container. The state names are unchanged but the CONNECTING and SYNCING stages have fundamentally changed: there are no WebSocket client managers to connect in the strategy container (those live in MarketDataService). CONNECTING now refers to constructing the RedisEventBusSubscriber, and SYNCING includes sync_from_exchange() + start_listener(channels) to activate the Redis subscription before the main loop begins.

Changes from Previous StrategyRunner

  • CONNECTING: now constructs RedisEventBusSubscriber instead of creating WebSocket managers
  • SYNCING: setup_pipeline() + setup_strategy() + start_listener(channels)
  • RUNNING: asyncio.gather(redis_listener_task, strategy.run_forever()) — two coroutines only
  • No feed tasks: removed run_multiplex_feeds() tasks — those run in MarketDataService container
  • Single strategy: reads STRATEGY_CONFIG_YAML env var; runs exactly one strategy per container

MarketDataService Lifecycle NEW

elysian_core/services/market_data_service.py  |  elysian_core/run_market_data.py

Overview

MarketDataService is a new dedicated service that owns all WebSocket feed connections. It runs as elysian_market_data in Docker and is a prerequisite for all strategy containers (via depends_on: condition: service_healthy). It has no strategies, no sub-account exchanges, no risk/execution components, and no Postgres connection (SKIP_DB_INIT=true).

Pure publisher — no subscriber logic: MarketDataService only publishes to Redis via RedisEventBusPublisher. It never subscribes to Redis, has no strategy callbacks, and does not consume any events it produces. The publish() method on the publisher is a simple Redis PUBLISH call.

RedisEventBusSubscriber — Listener Loop NEW

elysian_core/transport/redis_event_bus.py

Overview

The RedisEventBusSubscriber replaces the in-process EventBus as the shared market-data bus inside strategy containers. Its _listener_loop() runs as a persistent background asyncio.Task and is the primary source of KlineEvent and OrderBookUpdateEvent for the strategy. Backpressure is preserved because callbacks are awaited sequentially — a slow strategy hook stalls the listener loop, just as the original in-process EventBus did.

Backpressure preserved: Redis pub/sub messages are consumed one at a time from ps.listen(). Each message's callbacks are awaited sequentially before the next message is consumed. This matches the original in-process EventBus backpressure semantics — a slow on_kline() slows message consumption from Redis.

ActiveLimitOrder — Balance Reservation Lifecycle

elysian_core/core/order.py  |  elysian_core/core/shadow_book.py

Overview

When a LIMIT order is submitted, the ShadowBook locks the required capital before the order reaches the exchange. This prevents over-commitment when multiple strategies share the same account. For BUY orders, USDT (quote) is locked as reserved_cash. For SELL orders, the base asset quantity is locked as reserved_qty in the position's locked_quantity. Reservations are released incrementally as fills arrive and fully released when the order reaches a terminal state.

Capital safety: The reservation system prevents two strategies from both seeing the same free cash and each placing orders that together exceed available capital. The lock is placed synchronously inside execute() before the function returns, immediately after the exchange acknowledges the order ID.
BUY lock
reserved_cash = quantity × price (USDT locked)
SELL lock
reserved_qty = quantity (base asset locked)
Partial release
release_partial(delta_filled) on each fill event
Full release
release_all() on terminal order (FILLED / CANCELLED / EXPIRED)

ShadowBook — Derived State Refresh UPDATED

elysian_core/core/shadow_book.py

Overview

The ShadowBook maintains a locally consistent mirror of the account state. This is not a traditional FSM but a reactive computed-state pattern: whenever any input changes, _refresh_derived() is called to recompute all derived quantities atomically. Two new fields have been added: _asset_symbol_map and _position_deducted_comm to support precise non-stablecoin commission tracking. The start() method wiring has also changed — ORDER_UPDATE and BALANCE_UPDATE are no longer subscribed internally; they are dispatched by the strategy via asyncio.gather().

Changes from Previous ShadowBook

  • start() signature: now requires both event_bus (shared) and private_event_bus explicitly; KLINE subscribed internally, ORDER+BALANCE dispatched externally
  • _asset_symbol_map: new field mapping base asset (e.g. "BNB") → trading symbol (e.g. "BNBUSDT"); populated in apply_fill()
  • _position_deducted_comm: new field tracking non-stablecoin commission quantities already deducted from positions; excluded from total_commission() to avoid double-counting in net_pnl()
  • _klines dict: new field storing latest Kline per symbol (updated in _on_kline())
  • avg_entry_price patching: positions synced at startup with avg_entry_price=0.0 are patched on first KLINE event to use mark price, avoiding inflated unrealized PnL
  • apply_fill signature: now takes base_asset and quote_asset explicitly (required for commission mapping)

Commission Tracking — Non-Stablecoin Fees

Non-stablecoin commissions (e.g. BNB fees on BNBUSDT trades) require special handling. The _track_commission() method resolves the commission asset to a trading symbol via _asset_symbol_map and deducts the commission quantity from the position, mirroring what actually happens on the exchange. The deducted amount is recorded in _position_deducted_comm so it is excluded from total_commission() (which converts to USDT) — preventing double-counting in net_pnl().

# apply_fill() commission flow if commission_asset not in STABLECOINS: # Non-stablecoin fee: deduct from the position comm_symbol = _asset_symbol_map.get(commission_asset) if comm_symbol in _positions: _positions[comm_symbol].quantity -= commission _position_deducted_comm[commission_asset] += commission else: raise ValueError(f"Commission asset not found in positions") else: # Stablecoin fee: deduct from cash _cash -= commission _cash_dict[commission_asset] -= commission # total_commission() excludes position-deducted amounts def total_commission(self, mark_prices=None) -> float: for asset, gross_qty in _total_commission_dict.items(): net_qty = max(0.0, gross_qty - _position_deducted_comm.get(asset, 0.0)) # Only net_qty × price contributes to reported commission

BaseFSM — Core State Machine Engine

elysian_core/core/fsm.py

Overview

BaseFSM is the shared infrastructure that both OrderFSM and RebalanceFSM (and any future FSMs) are built on. It provides a declarative transition table, a re-entrancy guard, terminal state enforcement, wildcard trigger support, and both sync and async callback dispatch.

Re-entrancy guard: If a callback triggers a further trigger() call while a transition is already executing (the _transitioning flag is set), the new trigger is queued onto _pending (a deque) and drained after the current transition completes. This prevents lost events without needing locks.
Transition table
Dict[(State, trigger_str), (NextState, callback_name)]
Wildcard key
("*", trigger_str) matches any source state
Async callbacks
Both sync and async callbacks supported via await
Terminal states

UTIL 10

PeriodicTask — Background Scheduler Lifecycle

elysian_core/core/fsm.py  |  elysian_core/run_strategy.py (usage)

Overview

PeriodicTask is a lightweight asyncio utility that runs a callback on a fixed interval. It is used in the StrategyRunner to schedule save_snapshot() for both the Portfolio and each ShadowBook at regular intervals. The internal loop sleeps first, then calls the callback, ensuring the first invocation is delayed by one full interval from startup.

Primary usages
portfolio.save_snapshot(); shadow_book.save_snapshot()
Start
start() creates asyncio Task wrapping _loop()
Stop
stop() calls task.cancel() — clean cancellation
Loop order
while True: sleep(interval_s) → callback()
