FSM 1

OrderFSM — Order Lifecycle

elysian_core/core/order_fsm.py

Overview

The OrderFSM manages the complete lifecycle of a single exchange order from submission through to a terminal state. It operates in warn-mode: invalid transitions are logged but still applied, because the exchange is the authoritative source of truth for order state. The OrderFSM class wraps a _VALID_TRANSITIONS dict and is instantiated per-order. Module-level functions validate_order_transition() and is_terminal() are provided for backward compatibility.

Warn-mode: If an unexpected state transition arrives from the exchange feed (e.g. FILLED arriving on a CANCELLED order), the FSM logs a warning and still applies the transition rather than raising. The exchange is always authoritative.
Active states
PENDING, OPEN, PARTIALLY_FILLED, TRIGGERED
Terminal states
FILLED, CANCELLED, REJECTED, EXPIRED
Key property
is_active, can_fill, is_conditional, is_terminal
Fill tracking
sync_from_event() via _prev_filled delta

Implementation Notes

ActiveLimitOrder extends ActiveOrder and adds reservation management for capital locking. The sync_from_event() method on ActiveOrder tracks fill deltas via the _prev_filled field, ensuring each partial fill is processed exactly once even if exchange events arrive out of order or duplicated.

RebalanceFSM — Rebalance Cycle

elysian_core/core/rebalance_fsm.py

Overview

The RebalanceFSM drives the full signal-to-execution pipeline. Starting from IDLE, it orchestrates weight computation, validation, execution, and cooldown in a strict ordered cycle. A wildcard "suspend" trigger from any state enters SUSPENDED, implementing a drawdown breaker or maintenance pause.

Re-entrancy guard: BaseFSM queues transitions that arrive while one is already in-progress using a deque. This prevents lost events when callbacks trigger further transitions synchronously.
Pipeline stages
IDLE → COMPUTING → VALIDATING → EXECUTING → COOLDOWN
Error recovery
ERROR + "retry" → IDLE (auto-retry after cooldown_s)
Suspend trigger
Wildcard from any state → SUSPENDED
Cooldown
asyncio.call_later(cooldown_s); immediate if 0

StrategyState — Strategy Lifecycle

elysian_core/core/enums.py  |  elysian_core/strategy/base_strategy.py

Overview

StrategyState tracks the lifecycle of a running trading strategy from creation through to graceful shutdown. The STARTING state is where all EventBus subscriptions are registered — five event types across the shared and private buses. The strategy transitions from READY to RUNNING only when the first live event is dispatched, ensuring no compute runs before data is flowing.

FAILED transition: Any unhandled exception during the lifecycle — including during on_start(), event dispatch, or on_stop() — moves the strategy to FAILED from whatever state it is currently in.
Subscriptions (STARTING)
KLINE, ORDERBOOK_UPDATE, ORDER_UPDATE, BALANCE_UPDATE, REBALANCE_COMPLETE
RebalanceFSM init
Created in STARTING if optimizer + execution_engine are wired
Shutdown
executor shutdown + shadow_book.stop() in on_stop()
RUNNING trigger
_mark_running() on first dispatched event

RunnerState — StrategyRunner Lifecycle

elysian_core/core/enums.py  |  elysian_core/run_strategy.py

Overview

RunnerState represents the top-level lifecycle of the StrategyRunner process. It spans the entire operational lifecycle: from initial configuration loading through WebSocket connectivity, live data synchronisation, and into steady-state running. Shutdown is cleanly ordered — strategies are stopped, then feeds closed, then the executor shut down. Any fatal error at any stage moves directly to FAILED.

SYNCING stage: exchange.sync_from_exchange() is called here to hydrate the ShadowBook with live positions, open orders, and current balances before any strategy is started. This ensures strategies begin with accurate state.

SUB-FSM 5

ActiveLimitOrder — Balance Reservation Lifecycle

elysian_core/core/order.py  |  elysian_core/core/shadow_book.py

Overview

When a LIMIT order is submitted, the ShadowBook locks the required capital before the order reaches the exchange. This prevents over-commitment when multiple strategies share the same account. For BUY orders, USDT (quote) is locked as reserved_cash. For SELL orders, the base asset quantity is locked as reserved_qty in the position's locked_quantity. Reservations are released incrementally as fills arrive and fully released when the order reaches a terminal state.

Capital safety: The reservation system prevents two strategies from both seeing the same free cash and each placing orders that together exceed available capital. The lock is placed synchronously before the order is sent to the exchange.
BUY lock
reserved_cash = quantity × price (USDT locked)
SELL lock
reserved_qty = quantity (base asset locked)
Partial release
release_partial(delta_filled) on each fill event
Full release
release_all() when order is terminal

ShadowBook — Derived State Refresh

elysian_core/core/shadow_book.py

Overview

The ShadowBook maintains a locally consistent mirror of the account state — positions, cash, mark prices, NAV, weights, peak equity, and max drawdown. This is not a traditional FSM but a reactive computed-state pattern: whenever any input changes, _refresh_derived() is called to recompute all derived quantities atomically. This ensures the strategy always has a consistent, up-to-date portfolio snapshot without waiting for a full exchange round-trip.

Atomicity: _refresh_derived() recomputes NAV, weights, peak equity, and max drawdown in one synchronous pass. Any caller that holds a reference to the ShadowBook snapshot will see a consistent state.

BaseFSM — Core State Machine Engine

elysian_core/core/fsm.py

Overview

BaseFSM is the shared infrastructure that both OrderFSM and RebalanceFSM (and any future FSMs) are built on. It provides a declarative transition table, a re-entrancy guard, terminal state enforcement, wildcard trigger support, and both sync and async callback dispatch.

Re-entrancy guard: If a callback triggers a further trigger() call while a transition is already executing (the _transitioning flag is set), the new trigger is queued onto _pending (a deque) and drained after the current transition completes. This prevents lost events without needing locks.
Transition table
Dict[(State, trigger_str), (NextState, callback_name)]
Wildcard key
("*", trigger_str) matches any source state
Async callbacks
Both sync and async callbacks supported via await
Terminal states
Raise InvalidTransition on any further trigger

UTIL 8

PeriodicTask — Background Scheduler Lifecycle

elysian_core/run_strategy.py (usage)

Overview

PeriodicTask is a lightweight asyncio utility that runs a callback on a fixed interval. It is used in the StrategyRunner to schedule save_snapshot() at regular intervals, persisting portfolio state to disk without blocking the event loop. The internal loop sleeps first, then calls the callback, ensuring the first invocation is delayed by one full interval from startup.

Primary usage
PeriodicTask(save_snapshot, interval_s=N)
Start
start() creates asyncio Task wrapping _loop()
Stop
stop() calls task.cancel() — clean cancellation
Loop order
while True: sleep(interval_s) → callback()

State Diagram

stateDiagram-v2 direction LR [*] --> STOPPED : instantiated STOPPED --> RUNNING : start()\ncreate_task(_loop()) state RUNNING { direction TB [*] --> Sleeping Sleeping --> Executing : sleep(interval_s) elapsed Executing --> Sleeping : callback() returned } RUNNING --> STOPPED : stop()\ntask.cancel() note right of RUNNING _loop(): while True: await asyncio.sleep(interval_s) await callback() Cancellation raises CancelledError which exits the loop cleanly. end note

Interaction with RunnerState

The PeriodicTask is started in the SYNCING → RUNNING transition of the RunnerState FSM. It is stopped during the RUNNING → STOPPING transition, before strategies are stopped. This ordering guarantees that a final snapshot is saved after the last live events are processed but before the executor is torn down.

# Typical usage in StrategyRunner self._snapshot_task = PeriodicTask( callback=self._save_snapshot, interval_s=300 # every 5 minutes ) # Started in SYNCING → RUNNING await self._snapshot_task.start() # Stopped in RUNNING → STOPPING await self._snapshot_task.stop()
