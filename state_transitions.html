FSM 1

OrderFSM — Order Lifecycle

elysian_core/core/order_fsm.py  |  elysian_core/core/order.py

Overview

The OrderFSM manages the complete lifecycle of a single exchange order. It operates in warn-mode: invalid transitions are logged but still applied because the exchange is the authoritative source of truth. The same FSM is used for both SPOT and MARGIN orders — ActiveLimitOrder carries reservation state for both. Margin orders with AUTO_BORROW_REPAY pass through the same PENDING → OPEN → FILLED flow; the borrow/repay is invisible at the FSM level.

Active states
PENDING, OPEN, PARTIALLY_FILLED, TRIGGERED
Terminal states
FILLED, CANCELLED, REJECTED, EXPIRED
Margin note
MARGIN orders: same FSM. AUTO_BORROW_REPAY is transparent to the order lifecycle.
Fill tracking
sync_from_event() via _prev_filled delta

RebalanceFSM — Rebalance Cycle

elysian_core/core/rebalance_fsm.py

Overview & Margin Notes

The RebalanceFSM is instantiated once per sub-account pipeline in margin strategies (one FSM per SubAccountPipeline). Each FSM carries its own sub_account_key which is injected into ctx in _on_enter_computing() so compute_weights() can identify which pipeline is requesting weights. The FSM is otherwise identical in structure for SPOT and MARGIN pipelines — the differences are in the optimizer (margin level pre-check) and the execution engine (side_effect kwarg).

Per-pipeline FSM for MarginStrategy: MarginStrategy.start() creates one RebalanceFSM per pipeline: pipeline.rebalance_fsm = RebalanceFSM(strategy=self, optimizer=pipeline.optimizer, execution_engine=pipeline.execution_engine, sub_account_key=key). The strategy's request_rebalance(sub_account_key=key) targets a specific pipeline's FSM. Calling request_rebalance() with no key triggers all pipelines concurrently via asyncio.gather(*[p.rebalance_fsm.request() for p in _pipelines.values()]).
Pipeline stages
IDLE → COMPUTING → VALIDATING → EXECUTING → COOLDOWN
Margin: margin level check
VALIDATING: PortfolioOptimizer calls MarginShadowBook.margin_level() before constraint pipeline
sub_account_key injection
_on_enter_computing injects ctx['sub_account_key'] for MarginStrategy pipelines
Concurrent FSMs
MarginStrategy.request_rebalance() with no key → asyncio.gather all pipeline FSMs

StrategyState — Strategy Lifecycle UPDATED

elysian_core/core/enums.py  |  elysian_core/strategy/base_strategy.py  |  elysian_core/strategy/margin_strategy.py

Overview

Both SpotStrategy and MarginStrategy use the same StrategyState enum. The transition logic in start() differs: MarginStrategy.start() creates per-pipeline FSMs and subscribes to each pipeline's private bus, while SpotStrategy.start() creates one FSM and subscribes to one private bus.

MarginStrategy.start() vs SpotStrategy.start()

  • Shared bus: subscribed once (KLINE + OB) — same for both
  • Private bus: SpotStrategy subscribes 1 private bus; MarginStrategy subscribes N private buses (one per pipeline)
  • RebalanceFSM: SpotStrategy creates 1; MarginStrategy creates 1 per pipeline in _pipelines
  • _symbols: MarginStrategy collects union of all pipeline symbols
  • venues: MarginStrategy collects union of all pipeline venues
  • stop(): MarginStrategy unsubscribes each private bus; stops each shadow_book

RunnerState — StrategyRunner Lifecycle UPDATED

elysian_core/core/enums.py  |  elysian_core/run_strategy.py  |  elysian_core/run_strategy_margin.py

Overview

The same RunnerState enum is used by both StrategyRunner and MarginStrategyRunner. The SYNCING stage differs: margin runner builds one SubAccountPipeline per sub-account entry, while the standard runner builds one pipeline for the whole strategy. The RUNNING stage is identical — both gather the Redis listener task and run_forever().

MarginStrategyRunner differences from StrategyRunner

  • SYNCING: iterates strat_cfg.sub_accounts; calls _build_margin_pipeline(sub_cfg) per entry
  • Exchange type: selects BinanceMarginExchange vs BinanceSpotExchange based on sub_cfg.asset_type
  • ShadowBook type: selects MarginShadowBook vs ShadowBook based on sub_cfg.asset_type
  • Risk config: uses sub_cfg.risk_config (sub-account level) with fallback to effective_risk_for()
  • _compute_channels(): instance method override; maps MARGIN → SPOT channel format
  • Cleanup: calls exchange.cleanup() for each sub-account exchange (keyed by {strategy_id}_{sub_account_id})

SUB-FSM 7

ActiveLimitOrder — Balance Reservation Lifecycle

elysian_core/core/order.py  |  elysian_core/core/shadow_book.py  |  elysian_core/core/margin_shadow_book.py

Overview

When a LIMIT order is submitted, the ShadowBook (or MarginShadowBook) locks capital immediately. For SPOT BUY: locks USDT as reserved_cash → _locked_cash. For SPOT SELL: locks base asset as reserved_qty → position.locked_quantity. For MARGIN SELL (short entry): same as SPOT SELL pattern — reserved_qty on any existing long position, or if none, the lock is nominal since AUTO_BORROW_REPAY means Binance will borrow the base asset. The reservation is released incrementally as fills arrive.

BUY lock (SPOT)
reserved_cash = qty × price → _locked_cash += reserved_cash
SELL lock (SPOT/MARGIN)
reserved_qty = qty → position.locked_quantity += qty
Partial release
release_partial(delta_filled) on each fill event
Full release
release_all() on terminal order — drains remaining lock

MarginStrategy — Full Lifecycle State Machine MARGIN NEW

elysian_core/strategy/margin_strategy.py

Overview

MarginStrategy extends SpotStrategy and adds multi-pipeline management. It overrides start(), stop(), request_rebalance(), _dispatch_order(), and _dispatch_balance() to route events to the correct pipeline. The _shadow_book property is a backward-compatibility shim that returns the first pipeline's ShadowBook — used by RebalanceFSM logging only.

_pipelines dict is populated BEFORE start() is called: MarginStrategyRunner calls _build_margin_pipeline() for each sub-account, stores the results in strategy._pipelines, then calls strategy.start(). Inside start(), FSMs are created from the already-populated pipelines. This ordering is mandatory — FSMs require references to pipeline components.
_pipelines
Dict[SubAccountKey, SubAccountPipeline] — populated by runner before start()
_shadow_book property
Returns first pipeline's ShadowBook (backward compat only — read-only, no setter has effect)
request_rebalance(key=None)
key=None → asyncio.gather all FSMs. key=SubAccountKey → target 1 FSM.
shadow_books property
Dict[SubAccountKey, ShadowBook] — read per-pipeline book for strategy logic

BinanceMarginUserDataClientManager — Reader Loop MARGIN NEW

elysian_core/connectors/binance/BinanceMarginDataConnectors.py

Overview

Unlike BinanceUserDataClientManager (which uses a single global WebSocket via userDataStream.subscribe.signature), the margin manager uses a per-symbol listen key approach. Each isolated margin symbol gets its own POST /sapi/v1/userDataStream/isolated call and its own WebSocket reader task. A single keepalive task covers all symbols.

Reader tasks
One asyncio Task per symbol (_reader_tasks dict)
Listen key endpoint
POST/PUT/DELETE /sapi/v1/userDataStream/isolated?symbol=X&listenKey=K
WS URL
wss://stream.binance.com:9443/ws/{listenKey}
Keepalive
One task for all symbols; PUT every 55 min (expires at 60 min)
Event tagging
OrderUpdateEvent(asset_type=AssetType.MARGIN, sub_account_id=N)
Reconnect
Exponential backoff 1s → 60s; fresh listen key on each reconnect

MARGIN 14

Margin Poll Loop — Background State Sync MARGIN NEW

elysian_core/connectors/base_margin.py  |  elysian_core/connectors/binance/BinanceMarginExchange.py

Overview

The margin poll loop is the only mechanism available for keeping _accrued_interest current. Binance charges interest hourly on borrowed amounts but does not push these charges via the user-data WebSocket stream. Without periodic polling, the MarginShadowBook NAV would overstate value between interest cycles. The loop is started by BinanceMarginExchange.run() via _start_margin_poll() and stopped by cleanup() via _stop_margin_poll().

Interest accrual gap: Between poll cycles (up to 60 seconds), the MarginShadowBook may overstate NAV by the amount of accrued interest since the last poll. For typical margin sizes and rates, this is negligible (Binance daily rates are 0.02–0.05% annually for BTC/ETH). For strategies sensitive to sub-second NAV accuracy, reduce INTEREST_POLL_INTERVAL_S in IsolatedMarginExchangeConnector.
