ELYSIAN State Transitions

Every state machine, transition, trigger, callback, and terminal state in the system

1. BaseFSM — Generic Async Finite-State Machine Foundation for all FSMs

BaseFSM is the abstract base class for all state machines in Elysian. It provides a dict-based transition table, wildcard transitions, a re-entrancy guard, and optional EventBus integration. Subclasses define _TRANSITIONS as a mapping from (State, trigger_name) → (NextState, callback_method_name_or_None).

core/fsm.py — class BaseFSM, class PeriodicTask, class InvalidTransition

Core Mechanics

FeatureMechanismDetails
Transition Table _TRANSITIONS: Dict[(State, str), (State, Optional[str])] Class-level dict mapping (current_state, trigger_name) to (next_state, callback_name). Callback name is a string referencing a method on the subclass, or None to skip.
Wildcard Transitions ("*", trigger_name) Use the string "*" as the state key to match any current state. Explicit (State, trigger) matches take priority over wildcard matches. Used by RebalanceFSM for the "suspend" trigger.
Re-entrancy Guard _transitioning flag + _pending deque If a callback triggers another transition (e.g. _on_enter_computing triggers "weights_ready"), the nested trigger is queued in _pending rather than recursing. After the current transition completes, queued transitions are drained in FIFO order.
Terminal States _terminal_states: Set[Enum] States in this set reject all further transitions by raising InvalidTransition. Checked before table lookup.
EventBus Integration Optional _event_bus parameter When provided, subclass callbacks can publish events on state changes for external observability.
Context Passing **ctx keyword args Arbitrary context passed through trigger(event, **ctx) is forwarded to the callback method. The FSM also injects _fsm_old_state into ctx so callbacks can see the pre-transition state.

Key Methods

MethodSignaturePurpose
trigger async trigger(event: str, **ctx) -> Enum Fire a named trigger. Resolves transition from _TRANSITIONS, updates _state, runs the callback (sync or async), drains pending queue. Returns new state. Raises InvalidTransition if no matching transition exists.
_do_transition async _do_transition(event: str, **ctx) -> Enum Internal: resolves transition key (explicit first, then wildcard), updates _state, calls callback via getattr. Logs transition at DEBUG level.
state @property state -> Enum Returns the current state enum value.
is_terminal @property is_terminal -> bool True if current state is in _terminal_states.
Transition Resolution Order: 1. Check is_terminal — if True, raise InvalidTransition.
2. Look up (current_state, event) in _TRANSITIONS.
3. If not found, look up ("*", event) (wildcard).
4. If neither found, raise InvalidTransition.
5. Update _state to next_state.
6. Call callback (if not None) with **ctx.

2. RebalanceFSM — Rebalance Cycle State Machine Drives Stages 3-5

The RebalanceFSM (subclass of BaseFSM) replaces while True + asyncio.sleep() loops with an explicit state machine driving the compute → validate → execute → cooldown pipeline. It can be driven by a periodic timer (start_timer(interval_s)) and/or on-demand signals (request() / strategy.request_rebalance()).

Timer Mechanism

Timer Flow: start_timer(interval_s) → _schedule_tick() → call_later(interval, _fire_tick) → _fire_tick() checks if IDLE:
  • If IDLE: creates task _tick() which calls trigger("timer_tick")
  • If NOT IDLE: logs "Skipping tick — still in {state}", calls _schedule_tick() again
After cooldown or error recovery, _schedule_tick() is called again to restart the cycle.

Manual requests: strategy.request_rebalance() → RebalanceFSM.request() → checks if state == IDLE, then calls trigger("signal"). Returns False if busy.

3. OrderStatus FSM — Order Lifecycle Validation Warn-mode, exchange-authoritative

The Order FSM validates OrderStatus transitions but operates in warn-mode — illegal transitions are logged as warnings but still applied, because the exchange is the source of truth. This is not a BaseFSM subclass; it uses a simple dict of valid transitions and two standalone functions.

Order.update_fill(filled_qty, fill_price): Updates avg_fill_price using weighted average, updates filled_qty, and auto-transitions to FILLED if filled_qty >= quantity, or PARTIALLY_FILLED otherwise.

4. StrategyState — SpotStrategy Lifecycle 6-state linear progression

The SpotStrategy base class tracks its lifecycle via the StrategyState enum. Transitions are managed implicitly (not via BaseFSM) through guard clauses in start(), stop(), and _mark_running(). The state prevents double-start, out-of-order operations, and misuse of the pipeline.
